# Encryption Policy

**Company:** AcmeCloud Analytics  
**Product:** AcmeCloud Analytics cloud analytics platform for business reporting  
**Document Type:** Synthetic internal policy for demo use  
**Version:** 1.0  
**Effective Date:** 2026-01-15  
**Owner:** Platform Security Team  

## 1. Purpose

This Encryption Policy describes the fictional encryption controls used by AcmeCloud Analytics to protect customer data, internal service data, and backups for its cloud analytics platform. The policy is intended for a public-safe RAG demonstration and does not represent a real company, vendor environment, audit, or certification.

The objective of this policy is to reduce the risk of unauthorized data disclosure by requiring encryption for sensitive data at rest and in transit.

## 2. Scope

This policy applies to:

- Customer data stored by the AcmeCloud Analytics platform.
- Application databases used for reporting workspaces.
- File-based exports generated by the platform.
- Platform backups.
- Internal service-to-service communication where customer data or restricted operational data is transmitted.
- Employee administrative access to production systems.

This policy does not define encryption controls for customer-managed systems outside AcmeCloud Analytics. Customers are responsible for protecting files after they download or export them from the platform.

## 3. Data at Rest

Customer data stored by AcmeCloud Analytics is encrypted at rest using AES-256. This includes production application databases, object storage used for report assets, and internal storage used by the analytics processing layer.

AcmeCloud Analytics classifies stored customer data as restricted data. Restricted data must not be stored on employee laptops except through approved temporary support procedures. If a support export is required for troubleshooting, the export must be approved by the support manager and deleted when the troubleshooting task is complete.

Backups are also encrypted. Backup encryption is required for scheduled production backups, restoration test copies, and archived backup snapshots. Backup encryption settings are reviewed during the quarterly backup restoration test.

## 4. Data in Transit

Data in transit uses TLS 1.2 or higher. This requirement applies to customer browser sessions, API connections, administrative access to production consoles, and supported service-to-service communication.

AcmeCloud Analytics does not intentionally support plaintext access to the production application. If an internal service cannot support TLS 1.2 or higher, the Platform Security Team must document the exception, risk, compensating control, and remediation date.

External customer access to the application is intended to use secure HTTPS endpoints. The exact cipher suite list is not included in this evidence set. A question about specific cipher suites, certificate pinning, or exact TLS configuration would require human review or additional technical evidence.

## 5. Key Management

Encryption keys are managed through the company’s approved key management process. Production encryption keys must not be stored in source code repositories, plaintext configuration files, or employee notes. Access to key management functions is restricted to approved platform administrators and security operations personnel.

Key access is role-based and reviewed quarterly as part of the production access review process. Administrative access requires MFA. Key rotation is performed when a key is suspected to be exposed, when a system owner requests rotation for a defined maintenance event, or when required by an approved internal change plan.

This policy does not specify a fixed automatic key rotation interval for all encryption keys. Therefore, answers about annual, semiannual, or monthly rotation should be treated as not specified in the provided evidence unless supported by another document.

## 6. Secrets and Credentials

Application secrets, database credentials, API tokens, and signing secrets must be stored in approved secret storage mechanisms. Secrets must not be committed to source code repositories. If a secret is discovered in a repository, the finding must be reported to Security Operations and the secret must be rotated.

Service accounts must have an assigned owner and documented purpose. Access for service accounts must be limited to the minimum functions required for the service to operate.

## 7. Backups and Restoration

Backups are encrypted and tested quarterly. A quarterly backup test must include verification that a backup can be restored into a controlled non-production recovery environment. Test results must record the date of the test, system tested, responsible engineer, and whether the restore was successful.

Restored backup data used for testing must remain protected as restricted data. It must not be used for general training, development experiments, or public demonstrations.

## 8. Logging and Monitoring

Security-relevant encryption events may be logged where supported by the platform. These events may include failed administrative access attempts, changes to key permissions, and changes to encryption settings.

This policy does not specify whether every encryption-key use operation is logged. It also does not specify customer-visible encryption audit logs.

## 9. Customer Responsibilities

Customers are responsible for managing user access inside their own reporting workspaces, protecting downloaded reports, and configuring integrations according to their internal security requirements.

## 10. Limitations

This document is synthetic and does not claim external certification, regulatory compliance, or completion of an independent security audit. It should be used only as fictional evidence for a trust-response-agent portfolio project.
